💭 THINK FIRST · ZERO TRUST FUNDAMENTALS

You're about to configure Zero Trust connectivity across internet access, private access, and digital experience monitoring. Before touching any setting, anchor yourself to the core model — it makes every configuration decision in the next 12 labs easier to reason about.

  • In a traditional perimeter model, a user on the corporate network can reach any internal server unless a firewall rule explicitly blocks it. In the Zero Trust model you're about to configure, what is the default posture — and what does it mean for how App Segments and Access Policies must be constructed?
    Q1 — MODEL ANSWER
    Zero Trust starts from a default-deny posture: no user can reach any resource unless there is an explicit policy that permits it. This is the inverse of a perimeter model, where implicit trust is granted once a user is "inside." The practical consequence is that every application you want to expose must be defined in an App Segment (specifying which servers and ports are reachable) and an Access Policy must explicitly grant a user group permission to that segment. If you skip either — a segment with no policy, or a policy with no segment — access is denied. This model also means App Connectors never open inbound firewall ports; they initiate outbound connections to the Zscaler cloud, so the internal network is never exposed even to enrolled users.
  • This lab environment has a dedicated tenant (your personal ZIA/ZPA/ZDX configuration space) and a Solutions Demo Center (a shared, pre-populated read-only environment). Why does separating these two make sense for a training lab — and when would you need to switch between them?
    Q2 — MODEL ANSWER
    The dedicated tenant gives you full admin rights in a sandboxed environment — you can create, modify, and break things without affecting anyone else's session. This is where all the configuration labs (ZPA App Connectors, Browser Access, AppProtection, etc.) happen. The SDC gives you a realistic, pre-populated environment with live data already flowing — things like ZDX performance history, DLP incidents, and QBR reports that would take weeks to accumulate in a fresh tenant. You switch to SDC for Labs 11 and 12 specifically because those labs need existing data to analyze and incidents to triage. You would also switch if you needed to demonstrate ZDX metrics to a stakeholder without polluting your own clean lab config. Think of dedicated = your sandbox for building; SDC = a realistic production-like environment for analysis.

About this Lab

In this lab you apply skills from eLearning by working inside the Zscaler remote lab environment. You complete a series of exercises that build proficiency in:

  • Configuring connectivity to the Zero Trust Exchange
  • Defining allow and deny policies for access through the Zero Trust Exchange
  • Using the administration interfaces to analyse traffic patterns and user experience

What You Will Configure

  • Provision an App Connector and configure Browser Access
  • Configure Log Streaming to a SIEM
  • Build ZIA Browser Isolation (Zero Trust Browser)
  • Configure Privileged Remote Access (PRA)
  • Configure AppProtection (inline WAF) in ZPA
  • Build Zero Trust Browser Isolation in ZPA
  • Configure Cloud Application Monitoring in ZDX
  • Configure alerts and initiate a Diagnostics session in ZDX
  • Automate provisioning using Zscaler OneAPI
  • Analyse data and view integrated reports and snapshots (SDC)
  • Manage incidents with Zscaler Workflow Automation (SDC)

Lab Setup

You have access to the Zero Trust Exchange services — ZIA, ZPA, and ZDX — plus a ZIdentity account and a Skytap pod. You configure ZIA, ZPA, and ZDX, and use Windows and Linux client VMs in Skytap to connect to the Zero Trust Exchange and test access and user experience.

Task 0.4 Lab Access Information ▾

Dedicated Lab Tenant Access

Each student is assigned a unique Student ID with a dedicated virtual environment hosted in Skytap, and a corresponding Student FQDN with a set of admin and test user accounts for Zscaler. Your lab access instructions contain:

  • Student ID — a unique identifier for your pod (e.g. zs000834)
  • Student FQDN — a unique domain for your pod (e.g. zs000834.labtenant.com)
  • Login name format: student@<Student FQDN>
  • Session Password — unique to your session, shared across all admin and user accounts in your pod

Save your Student FQDN and Session Password in the ⚙ Lab Configuration panel at the top of every lab page. All credential placeholders will update automatically across all steps.

Solutions Demo Center (SDC) Access

For Labs 11 and 12, you switch to the Solutions Demo Center — a shared read-only Zscaler tenant with pre-populated data including ZDX performance history, DLP incidents, and workflow automation templates. You have a read-only administrator account on the SDC tenant.

  1. Open a new browser window and go to https://www.console.zscaler.com
  2. Sign in using the SDC credentials provided to you through email (or your lab resources panel).

The SDC tenant is hosted in the zscalerthree.net cloud. Use the Admin Portal URL https://www.console.zscaler.com. You can also use your local workstation browser to access it if preferred.
