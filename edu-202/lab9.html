LAB 9 · ZDX

ZDX Alerts & Diagnostics

Monitoring collects data — but without automated alerting you still rely on users to report problems. In this lab you configure a ZDX alert rule that fires when application performance degrades beyond threshold, and launch a Diagnostic Session to capture real-time device telemetry including Wi-Fi, DNS, cloud path, and application metrics for a specific user.

Lab 9: ZDX Configure Alerts & Diagnostics

LAB 9

Scenario: You are a NOC Engineer and you are seeing a spike in complaints that an internal application intermittently freezes or takes too long to load for remote users. By the time IT investigates, performance often returns to normal — making it difficult to prove impact, identify patterns, or determine whether the issue is user-specific, site-specific, ISP-related, or application-related. To reduce mean time to detect (MTTD) and accelerate troubleshooting, you configure ZDX alerts to automatically notify the NOC when user experience degrades beyond acceptable thresholds, and launch a Diagnostics Session to capture real-time client and network telemetry.

Time: ~15 minutes
Service: Zscaler Digital Experience (ZDX)
Prerequisite: Lab 8 probes (Mail Server_Web, Mail Server_Cloud) created and enabled
💭 THINK FIRST · ZDX ALERTING & DIAGNOSTICS

ZDX alerts shift your NOC from reactive (waiting for users to call) to proactive (getting notified before users notice). Diagnostic Sessions collect deep telemetry on demand. Think about how these tools work together to reduce MTTD and MTTR.

  • You set "Alert Only if Repeated = 2 Times in a Row". Why is this setting important, and what problem does it solve compared to alerting on the very first degraded probe result?
    Q1 — MODEL ANSWER
    Probe results are sampled every 5 minutes and can vary due to transient conditions: a momentary DNS spike, a brief ISP blip, or the probe running during a GC pause on the application server. Alerting on the first failure would generate many false-positive pages for issues that self-resolve within one probe cycle. Requiring 2 consecutive failures (10 minutes of sustained degradation) filters out transient noise while still catching genuine ongoing incidents quickly enough to respond before users widely notice. This is the standard "flap detection" technique used in all monitoring systems.
  • A Diagnostic Session collects "Deep Tracing" telemetry: device resource health (CPU/memory), Wi-Fi signal and roaming details, DNS resolution performance, and application performance breakdown — all from the endpoint. How does this change the troubleshooting workflow compared to asking the user to run manual diagnostics?
    Q2 — MODEL ANSWER
    Manual diagnostics from a user require the user to: (1) know how to run tools like traceroute or nslookup, (2) do so while the problem is still active, (3) correctly interpret and report the output. This is unreliable — most users can't reproduce the problem on demand and provide actionable data. A ZDX Diagnostic Session automates all of this: the NOC engineer launches it centrally, specifying the user and duration, and ZCC on the endpoint immediately starts streaming granular telemetry back without any user action. The data covers exactly what the user is experiencing right now, and the session report visualises the full network path as a hop-by-hop diagram.
Task 9.1 Create an Alert Rule ▾

An alert rule monitors probe metrics and notifies the NOC automatically when thresholds are breached. You will create a rule for the Mail Server application that triggers when page fetch time is too high or the ZDX score drops too low.

  1. On Corp: Client PC 1, sign in to Zscaler Client Connector using your student credentials.
  2. Open a browser and navigate to the Experience Center. Go to Administration > Alerts > Digital Experience Management > Alerts.
  3. Select Alert Rules and Templates to navigate to the Rules page.
  4. Select Add Alert Rule.
  5. In the Add Alert Rule window, enter:
    • Name: alert_mailserver
    • Status: Enabled
    • Type: Application
    • Severity: Medium
  6. Select Next.
  7. On the Filters tab, configure the application filters:
    • Application: Mail Server (the application created in Lab 8)
    • Web Probe: Mail Server_Web
  8. Add a geolocation filter:
    • Select + Add Filter, then select Geolocation.
    • Expand Asia and select all countries in Asia.
    • Confirm the filter shows Geolocations Include 50 (or similar).
  9. Select Next.
  10. On the Criteria tab, create the following conditions:
    • Page Fetch Time >= 800 ms
    • ZDX Score < 50
  11. Select Next.
  12. On the Actions tab, enter:
    • Alert Only if Repeated: 2 times in a row
    • Number of Active Devices is: 1
    • Alert Delivery Method: Email
    • Alert recipient: Enter any dummy email address (e.g. noc@example.com)
  13. Select Next.
  14. On the Review tab, review the entire configuration and select Save.
  15. Verify: The alert_mailserver alert rule appears in the Rules list with Status = Enabled, Type = Applications, Application = Mail Server, Probe = Mail Server_Web, Alert Delivery Method = Email.

Task 9.2 Configure a Diagnostic Session ▾

A Diagnostics Session instructs the ZCC client on a specific user's device to run deep telemetry collection for a set duration. The session report provides a hop-by-hop cloud path diagram, device resource health, Wi-Fi metrics, DNS performance, and application response times — all timestamped.

  1. Navigate to Analytics > Digital Experience Management > Troubleshoot > Diagnostics.
  2. Select Start New Diagnostic Session.

    If "Start New Diagnostic Session" is not visible, you need to enable machine hostname collection first:

    1. Go to Infrastructure > Connector > Client > Supportability > User Privacy.
    2. Select the User Privacy tab.
    3. Enable Collect Machine Hostname Information.
    4. Refresh the Diagnostics page — the button should now appear.

  3. In the Start New Diagnostics Session window, enter:
    • Name: diagnostics_session_student<studentID> — e.g. diagnostics_session_student000205
    • User: Student Admin User Name assigned to you (student@<Student FQDN>)
    • Device: Select the device associated with your student user
    • What would you like to run: Select Deep Tracing
    • Run Session For: 5 Minutes
  4. Select Next.
  5. Select the probes for the session:
    • Application: Mail Server
    • Web Probe: Mail Server_Web
    • Cloud Path Probe: Mail Server_Cloud
  6. Under Cloud Path Probe Thresholds, set:
    • Packet Loss: 20%
    • Latency: 10 ms
  7. Select Save.
  8. Confirm the session appears in the In Progress table with Status = In Progress.
  9. Proceed to the next lab. After five minutes, return to Diagnostics.
    • In the History table, locate the completed session (Status = Completed).
    • Select the View icon to open the session report.
    • The report shows a Cloud Path hop-by-hop diagram: Client → Wi-Fi Access Point → Gateway → ZIA Public Service Edge → Application, with per-hop latency values and the selected thresholds shown as dotted red lines.

Expected result: The session report shows the full network path from the client device to mail.google.com. The Cloud Path diagram displays latency at each hop, allowing you to immediately pinpoint whether degradation is local (Wi-Fi/LAN), ISP-related, ZIA edge-related, or application-server-related without any user involvement.

MENTAL NOTE
  • Proactive Alerting: ZDX Alert Rules fire automatically when Page Fetch Time or ZDX Score breaches a threshold — shifting the NOC from reactive to proactive.
  • Noise Filtering: Alert Only if Repeated = 2 suppresses transient spikes and avoids alert fatigue.
  • Diagnostic Sessions: Extend monitoring from continuous synthetic probes to on-demand deep telemetry — a timestamped hop-by-hop breakdown of a specific user's path.
