LAB 6 · ZPA

ZPA AppProtection

AppProtection embeds a web application firewall into ZPA's data path, inspecting every private application request against OWASP CRS rules, ThreatLabZ intelligence, and protocol-specific controls. In this lab you protect the Damn Vulnerable Web App (DVWA) — deliberately insecure — and watch SQL injection and XSS attacks flow through in Report mode before switching to Block.

Lab 6: ZPA AppProtection

LAB 6

Scenario: You are a security engineer responsible for protecting internal web applications from common vulnerabilities such as SQL injection and cross-site scripting (XSS). You will configure Zscaler Private Access™ (ZPA) AppProtection to inspect traffic and enforce security controls for the Damn Vulnerable Web App (DVWA) — an intentionally insecure application used for security training.

Time: ~30 minutes
Service: Zscaler Private Access (ZPA) — AppProtection
Prerequisite: App Connector running and connected
💭 THINK FIRST · APPPROTECTION

AppProtection runs inside ZPA's proxy path between the user and the internal application. Before configuring it, think about how it differs architecturally from a traditional WAF and why starting in Report mode matters operationally.

  • A traditional WAF sits in front of a public-facing application. ZPA AppProtection sits between an authenticated ZPA session and a private application. What attack vectors does AppProtection cover that traditional WAFs typically cannot?
    Q1 — MODEL ANSWER
    Traditional WAFs protect public-facing apps from external attackers who may be anonymous. AppProtection protects private apps from authenticated internal users and trusted insiders — threats that never reach a public WAF. This includes: (1) insider threats and compromised employee accounts, (2) attacks via Browser Access sessions (e.g. contractors with limited access who try to exploit the internal app), (3) lateral-movement attacks from already-breached internal machines. AppProtection also adds identity context — it knows which ZPA user triggered an inspection event, which is something a traditional WAF cannot easily correlate.
  • OWASP CRS has four Paranoia Levels (1–4). You set level 3. What is the trade-off between a high and low paranoia level, and why might level 4 be problematic in production?
    Q2 — MODEL ANSWER
    Paranoia Level 1 activates only the highest-confidence OWASP rules — low false positives but may miss sophisticated attacks. Level 4 activates all 298+ rules including highly sensitive heuristics — catches more attacks but generates many false positives on legitimate traffic (e.g. URL parameters with special characters, REST APIs with JSON payloads). Level 3 is the practical balance: good coverage of real-world SQL injection and XSS patterns without excessive noise. Level 4 in production typically requires extensive tuning of Override Controls (specific rules set to Allow) before it's safe to move to Block mode.
  • You configure Override Controls with Alternate Action = "Override All Controls, Transaction = Allow" for all control groups. Why start in Report mode this way rather than Block, even though DVWA is known to be vulnerable?
    Q3 — MODEL ANSWER
    Even for a deliberately vulnerable app, starting in Report (Allow + log) mode is the right operational practice because: (1) it lets you see which rules fire on normal app usage before deciding what to block, (2) it avoids breaking the app during the lab exercise — blocking attacks against DVWA would prevent the SQL injection and XSS tests, defeating the purpose, (3) it demonstrates the standard "observe → tune → enforce" WAF deployment lifecycle. In production, you would deploy in Report for days or weeks, review the violation logs, add Override Controls for confirmed false positives, then move to Block mode rule-by-rule or all at once.
Task 6.1 Create DVWA App Segment with AppProtection ▾

DVWA runs in a Docker container at 10.0.0.128. Before creating the App Segment, you need to add a DNS A record on the Corp: Win Svr so the ZPA connector can resolve the app hostname.

Step 1 — Add DNS Record on Corp: Win Svr

  1. On Corp: Win Svr, open DNS Manager (Server Manager → Tools → DNS).
  2. Navigate to Forward Lookup Zones > patraining.safemarch.com.
  3. Right-click and select New Host (A or AAAA)…
  4. Configure:
    • Name: dvwa-<StudentID> — replace <StudentID> with the last three digits of your tenant ID (e.g. if your FQDN is <Student FQDN>, and your Student ID is <Student ID>, the hostname is dvwa-<last3digits>)
    • IP Address: 10.0.0.128
  5. Select Add Host, then Done.

Step 2 — Create the DVWA App Segment

  1. In the Experience Center, go to Policies > Access Control > Private Applications > App Segments.
  2. Select Add Application Segment and configure:
    • Name: DVWA
    • Applications: dvwa-<StudentID>.patraining.safemarch.com (the hostname you added to DNS)
    • Access Type: Check both Browser Access and AppProtection
    • Certificate Type: Managed
    • External URL prefix: dvwa
    • Internal URL: HTTP://dvwa-<StudentID>.patraining.safemarch.com:80
  3. Under Client Connector Access, configure Web Server ports (keep defaults for HTTP traffic).
  4. Select Next, choose Segment Group CorpApps, select Next, choose Server Group CorpServers, select Next.
  5. Review and select Save.
  6. Create the Access Policy rule for DVWA. Navigate to Policies > Access Policy and select Add:
    • Rule Name: dvwa_allow_policy
    • Rule Action: Allow Access
    • Under Criteria → Applications, select the DVWA app segment
  7. Select Save.

Note: The Browser Access URL for DVWA will be https://dvwa-<studentid>.labtenant.com.clientless.two.zscalerportal.net (or similar — it is shown in the App Segment details after saving). You will use this URL in Task 6.4.

Task 6.2 Create DVWA Protection Profile ▾

A Protection Profile defines which control groups are active and at what enforcement level. You will add all available controls and set Override Actions to Allow (Report mode) — attacks are logged but not blocked.

  1. Navigate to Policies > Cybersecurity > Inline Security > Private App Protection > Protection Profiles.
  2. Select Add Profile.
  3. Name: DVWA_Protection_Profile
  4. Add the following control groups by clicking + Add Controls for each:
    • OWASP CRS/4.8.0 — select All controls (298 rules). These cover SQL injection, XSS, command injection, path traversal, and more.
    • ThreatLabZ Controls — select All controls (~200 rules). Zscaler's threat intelligence-based signatures.
    • WebSocket Controls — select All controls (11 rules).
    • API Controls — select All controls.
    • Active Directory Controls — select all 4 controls: KRB (Kerberos), SMB, LDAP.
  5. Set Predefined Paranoia Level: 3.
  6. Configure Override Controls for each control group:
    • Alternate Action: Override All Controls
    • Transaction: Allow (this puts all controls in Report mode — violations are logged but traffic passes)
    • Apply this Override to: OWASP CRS, ThreatLabZ Controls, WebSocket Controls, API Controls, Active Directory Controls
  7. Select Save.

What "Paranoia Level 3" means: At level 3, OWASP CRS activates rules with moderate confidence — good for detecting real attacks while keeping false-positive rates manageable. Level 1 is most lenient; level 4 is most aggressive and requires significant tuning before blocking.

Task 6.3 Create DVWA AppProtection Policy ▾

The Protection Policy links the Protection Profile to the App Segment. Without this policy, AppProtection is defined but never applied to DVWA traffic.

  1. Navigate to Policies > Cybersecurity > Inline Security > Private App Protection > Protection Policies.
  2. Select Add Policy.
  3. Configure:
    • Name: DVWA_AppProtection_policy
    • Rule Action: Inspect
    • AppProtection Profile: DVWA_Protection_Profile
  4. Under Criteria, add the DVWA Application Segment.
  5. Select Save.

Rule Action: Inspect means ZPA runs traffic through the Protection Profile rules. The Profile's Override Controls (Allow) determine whether matching requests are blocked or merely logged. If you set Rule Action to Bypass, AppProtection would be skipped entirely for this traffic.

Task 6.4 Test DVWA — SQL Injection, XSS and Verify Violations ▾

Step 1 — Access DVWA via Browser Access

  1. Log out of Zscaler Client Connector on Corp: Client PC 1 to simulate a Browser Access session.
  2. Open Chrome and navigate to the DVWA Browser Access URL. You can find this URL in the App Segment details — it resembles https://dvwa-<studentid>.labtenant.com.clientless.two.zscalerportal.net.
  3. Log in to DVWA using:
    • Username: admin
    • Password: password
  4. On the DVWA home page, select Create / Reset Database to initialise the database. The page refreshes and you are prompted to log in again.
  5. Log in again with admin / password.

Step 2 — SQL Injection Test

  1. In the DVWA left menu, select SQL Injection.
  2. In the User ID field, enter the following payload and select Submit:
    1' or '1'='1
  3. DVWA displays multiple user records — the query was successfully injected. Because AppProtection is in Report (Allow) mode, the request passes through and ZPA logs the violation.

Step 3 — Stored XSS Test

  1. In the DVWA left menu, select XSS (Stored).
  2. In the Name field, enter: JSAttack
  3. In the Message field, enter:
    <script>alert("You've been HACKED!")</script>
  4. Select Sign Guestbook. The script tag is stored. When the page reloads, you may see the alert pop up (Report mode allows it through). ZPA logs the XSS violation.

Step 4 — Verify Violations in Admin Portal

  1. In the Experience Center, navigate to Logs > Insights > Private Applications > Diagnostics.
  2. Set Log Type: AppProtection.
  3. Confirm that Security Profile Violations appear — you should see entries for the SQL injection and XSS payloads, showing the matched rule name, severity, and that the transaction was Allowed (not Blocked).

Step 5 — Switch to Block Mode

  1. Return to Policies > Cybersecurity > Inline Security > Private App Protection > Protection Profiles.
  2. Edit DVWA_Protection_Profile.
  3. Update the Override Controls for each control group:
    • Change Transaction: from Allow to Block
  4. Select Save.
  5. Retry the SQL injection payload in DVWA — the request should now be blocked with an error page. Verify the block event appears in the AppProtection diagnostics logs.

Verify: In Block mode, the SQL injection payload 1' or '1'='1 should return a ZPA block page rather than database records. The AppProtection log entry will show Transaction = Blocked.
