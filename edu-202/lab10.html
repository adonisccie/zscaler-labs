Lab 10: Automation using Zscaler APIs

LAB 10

Scenario: Your organisation is expanding its Zero Trust deployment and needs to automate provisioning tasks that would otherwise require manual portal access. You are tasked with integrating Zscaler into your automation pipeline using the OneAPI platform. In this lab you set up Postman as your API client, import the official Zscaler OneAPI collection, configure OAuth 2.0 authentication, validate live API calls against ZCC device data, and create a new App Connector Group via a POST request — demonstrating the full API-driven provisioning workflow.

Time: ~20 minutes
Service: Zscaler OneAPI (ZIA · ZPA · ZCC · ZDX)
Tool: Postman (web or desktop)
💭 THINK FIRST · ZSCALER ONEAPI & AUTOMATION

OneAPI unifies ZIA, ZPA, ZCC, and ZDX under a single OAuth 2.0 authentication model. Before writing a single API call, make sure you understand why credentials are scoped and time-limited — and what "API Full Access" actually permits.

  • When you create an API Client in ZIdentity you must copy the Client Secret immediately — it cannot be viewed again after saving. Why does this design pattern exist in OAuth 2.0 systems, and what operational control does it enforce?
    Q1 — MODEL ANSWER
    This is a standard security design principle called "secret exposure minimisation". The system stores only a hashed form of the secret — never the plaintext — so it cannot be retrieved even by the platform itself. Showing the plaintext once at creation time forces the engineer to immediately store it in a secrets manager (Vault, AWS Secrets Manager, etc.) or CI/CD credential store. If it must be rotated, a new secret is generated and the old one is revoked. This means a compromised developer laptop with a screen recording of the creation wizard does not give an attacker indefinite API access — they only ever see the secret once, and the organisation can rotate it at any time without re-creating the entire API client.
  • The API client is configured with "Access Token Validity: 90 Minutes". An access token expires — what should a well-designed automation pipeline do when a request returns HTTP 401 Unauthorized, rather than failing permanently?
    Q2 — MODEL ANSWER
    A robust pipeline should implement an automatic token refresh loop: (1) attempt the API call with the current token; (2) on 401, re-authenticate using the Client ID and Client Secret to obtain a new access token; (3) retry the original request with the new token; (4) if re-authentication also fails (e.g. secret rotated), alert the operations team and halt. Short-lived tokens are a feature, not a bug — they limit blast radius if a token is intercepted (e.g. in a log file or network trace). The 90-minute window is a balance between security (short lifetime) and operational overhead (too-frequent re-authentication).
Task 10.1 Access and Set Up Postman ▾

Postman is the API client used in this lab. You can access it via the web browser or use the desktop application installed on your lab machine. A Postman account is required to save collections and variables between sessions.

  1. On Corp: Client PC 1, open a web browser and navigate to postman.com.

    Alternatively, use the Postman desktop application installed on the Corp: Client PC 1 desktop if it is available.

  2. If you do not already have an account, select Sign Up for Free and create a free Postman account.
  3. Sign in to Postman. Confirm you can see the Postman workspace home screen.

Task 10.2 Import the Zscaler OneAPI Postman Collection ▾

Zscaler publishes an official Postman collection for OneAPI that includes pre-built request templates for all product areas. You download it from the Zscaler developer portal and import it into your workspace.

  1. Open a new browser tab and navigate to automate.zscaler.com.
  2. Locate and download the Zscaler OneAPI Postman Collection file (a .json file).
  3. Return to Postman. Select File > Import (or use the Import button in the left-hand Collections sidebar).
  4. Drag and drop the downloaded .json file, or browse to it. Select Import.
  5. Verify that a collection named OneAPI appears in the Collections panel on the left, with sub-folders including:
    • ZIA (Zscaler Internet Access)
    • ZPA (Zscaler Private Access)
    • ZCC (Zscaler Client Connector)
    • ZDX (Zscaler Digital Experience)
    • ZIdentity

Expected result: The OneAPI collection is visible in Postman Collections with all product sub-folders expanded and accessible.

Task 10.3 Gather Variables for OneAPI Authentication ▾

OAuth 2.0 authentication requires four variables: the ZIdentity base URL, a Client ID, a Client Secret, and your ZPA Customer ID. You collect these from different areas of the administration portal.

Step A — Find the ZIdentity Base URL

  1. In the Experience Center, navigate to Administration > Identity > ZIdentity > ZIdentity Portal.
  2. The ZIdentity portal opens in a new browser tab. Copy the base URL from the browser address bar (e.g. https://zs000149.zslogin.net). This is your zidentityBaseUrl. Note it for later.

Step B — Create an API Client in ZIdentity

  1. In the Experience Center, navigate to Administration > API Configuration > OneAPI > API Clients.
  2. Select Add API Client. In the Add API Client dialog, enter:
    • Name: TestClient
    • Status: Enabled
    • Access Token Validity: 90 Minutes
  3. Under Client Authentication, set:
    • Validation Type: Secret
    • Select Add.
  4. Select the Resources tab. Grant the following permissions:
    • ZCC: Super Admin
    • ZIA: Super Admin
    • ZPA: API Full Access
  5. Select Save.
  6. Immediately copy the displayed Client Secret and store it securely.

    Important: The Client Secret is shown only once. Once you close or navigate away from this dialog it cannot be retrieved. If you miss it, you must regenerate the secret.

  7. Verify the TestClient API client appears in the API Clients list with its Client ID visible. Note the Client ID for later.

Step C — Find the ZPA Customer ID

  1. Navigate to Administration > Account Management > Linked Services.
  2. Locate the ZPA Tenant ID (also referred to as the Customer ID). Note this value for later.

Task 10.4 Authenticate and Test OneAPI Endpoints ▾

You configure the collected variables in the Postman collection, obtain an OAuth 2.0 access token, and then test two live ZCC API endpoints to verify connectivity and access.

Step A — Set Collection Variables

  1. In Postman, select the OneAPI collection in the left panel.
  2. Select the Variables tab.
  3. Update the following variables with the values you collected in Task 10.3:
    • zidentityBaseUrl — your ZIdentity portal URL (e.g. https://zs000149.zslogin.net)
    • clientSecret — the Client Secret copied when you created TestClient
    • clientID — the Client ID shown on the TestClient row
    • customerid — the ZPA Tenant/Customer ID from Linked Services
  4. Select Save (or press Ctrl+S / Cmd+S).

Step B — Obtain an Access Token

  1. With the OneAPI collection selected, select the Authorization tab.
  2. Scroll down to the bottom of the Authorization panel and select Get New Access Token.
  3. Postman performs the OAuth 2.0 flow against ZIdentity. When complete, a dialog displays "Authentication complete".
  4. Select Use Token to apply the token to the collection.

Step C — Verify the Active Token in ZIdentity

  1. In the Experience Center, navigate to Administration > API Configuration > OneAPI > Tokens.
  2. Filter or search by your Client ID (or filter by Last 15 minutes). Confirm an active token appears for TestClient.

Step D — Test ZCC Endpoints

  1. In Postman, navigate to OneAPI > Zscaler Client Connector.
  2. Select the request GET "Get all enrolled devices" and select Send.
    • Expected response: 200 OK
    • The response body contains a JSON array of enrolled devices.
  3. Select the request GET "Download device details as a CSV file" and select Send.
    • Expected response: 200 OK
    • The response body contains CSV-formatted device details including device name, OS version, ZCC version, and enrolled user.

Expected result: Both requests return 200 OK with valid response bodies. This confirms that OAuth 2.0 authentication is working and your API client has the correct ZCC Super Admin permissions.

Task 10.5 Create an App Connector Group via API ▾

You use a POST request to programmatically create a new ZPA App Connector Group — the same resource you would normally configure by hand through the ZPA portal. This demonstrates the full create workflow: look up a dependency (enrollment certificate ID), then issue a create request that references it.

Step A — Verify existing App Connector Groups

  1. In the Experience Center, navigate to Infrastructure > Private Access > App Connector Groups.
  2. Note the App Connector Groups currently configured. After completing this task, a new group named ZPALAB APPC Group should appear in this list.

Step B — Retrieve the Enrollment Certificate ID

  1. In Postman, navigate to OneAPI > ZPA > Enrollment Certificate Controller.
  2. Select the request GET "Gets all configured enrollment certificate details for specified customer" and select Send.
  3. In the JSON response body, locate the enrollment certificate and note its id value (e.g. "id": "7651"). You will use this as the signingCertId in the next step.

Step C — Create the App Connector Group

  1. In Postman, navigate to OneAPI > ZPA > Connector Group Controller.
  2. Select the request POST "Adds a new App Connector Group for specified customer".
  3. Select the Body tab. Replace the existing body with the following JSON, substituting the actual id value you retrieved in Step B for signingCertId: 
    {
  "name": "ZPALAB APPC Group",
  "description": "App Connector training Lab at San Jose",
  "latitude": "37.3382032",
  "longitude": "-121.886286",
  "location": "San Jose, CA, USA",
  "signingCertId": "7651"
}

    Replace "7651" with the actual id value returned from the enrollment certificate GET request in Step B.

  4. Select Send.
  5. Verify the response code is 201 Created. The response body contains the full JSON representation of the newly created App Connector Group.

Step D — Confirm in the ZPA Portal

  1. Return to the Experience Center and navigate to Infrastructure > Private Access > App Connector Groups.
  2. Confirm that ZPALAB APPC Group now appears in the list — created programmatically via the API without any portal interaction.

Expected result: The POST request returns 201 Created and the new ZPALAB APPC Group is immediately visible in the ZPA App Connector Groups portal. This completes the full OneAPI provisioning workflow: authenticate → query dependencies → create resource.

MENTAL NOTE
  • OAuth 2.0 Flow: Create an API Client in ZIdentity with scoped permissions, collect the one-time Client Secret, then authenticate to receive a time-limited access token.
  • Cross-Service Token: The same token authenticates against ZIA, ZPA, ZCC, and ZDX — one auth flow for the entire platform.
  • Two-Step Provisioning: GET a dependency first (e.g. enrollment certificate ID), then POST the new resource referencing that dependency.
  • Success Signal: A 201 Created response confirms provisioning — no portal interaction required.
