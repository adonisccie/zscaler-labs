LAB 12 · WORKFLOW AUTOMATION

Manage Incidents with Zscaler Workflow Automation

A DLP violation without a response workflow is just noise. In this lab you explore the Zscaler Workflow Automation (ZWA) portal: view DLP incidents, examine trigger data, update status and priority, notify the end user, escalate to an approver, and examine the automated workflow templates and mappings that drive consistent incident response — without manual triage for every alert.

Lab 12: Manage Incidents with Zscaler Workflow Automation (ZWA)

LAB 12

Scenario: Your organisation has DLP policies enforced through ZIA. When a violation occurs, a DLP incident is created in the Zscaler Workflow Automation (ZWA) portal. You are the security operations analyst responsible for triaging these incidents. In this lab you enroll a test device using SDC credentials, locate a DLP incident, examine its metadata and trigger content, update its status and priority, notify the originator, escalate to an approver, and explore the automated workflow templates that can eliminate manual triage steps entirely.

Time: ~20 minutes
Service: ZIA DLP · Zscaler Workflow Automation (ZWA)
Environment: Solutions Demo Center (SDC)
💭 THINK FIRST · INCIDENT MANAGEMENT & WORKFLOW AUTOMATION

A DLP incident is the enforcement layer's signal that a policy was triggered — but the incident itself is just a data point. The value comes from what happens next: classification, notification, escalation, and resolution. Before you work through the manual steps in this lab, consider what automation can replace.

  • When a DLP incident is set to "Investigating" and the originator is notified to provide justification, there is a gap: what if they never respond? What lifecycle states would you design into an automated workflow to ensure every incident reaches a terminal state (Resolved or Dismissed) without manual follow-up?
    Q1 — MODEL ANSWER
    A robust lifecycle would include: (1) New → auto-acknowledge after creation with a timestamp; (2) Investigating → notify user with a response deadline (e.g. 48 hours); (3) if no response → auto-escalate to the user's manager or the security team; (4) if escalation is not actioned within an SLA window → auto-resolve as "No Response — Suspicious" or create a high-priority ticket in the ITSM system; (5) if the user provides a valid justification → auto-close as Dismissed with the justification recorded. The goal is zero abandoned incidents: every alert either gets a human decision or triggers a defined automated consequence. Workflow templates in ZWA encode this logic so it applies consistently without analyst involvement on every routine incident.
  • Workflow Mappings define when a template triggers based on incident attributes (policy, severity, user department, etc.). Why is it important to match workflow templates to specific incident types rather than applying a single universal workflow to all DLP incidents?
    Q2 — MODEL ANSWER
    Different DLP violations carry different risk profiles and require different responses. A high-severity PCI data egress incident from the Finance department warrants immediate auto-escalation to the CISO and legal team with a 4-hour SLA. A low-severity keyword match from an intern might only warrant a coaching notification with no escalation. Applying a single heavy-handed workflow to all incidents would: (1) overwhelm approvers with low-severity noise; (2) delay critical incidents that need faster escalation; (3) create alert fatigue where analysts start ignoring notifications. Mapping specific templates to specific incident attributes (policy type, data classification, user department, severity) ensures proportional response — each incident gets the right amount of attention and automation, not a one-size-fits-all treatment.
Task 12.1 Enroll with Zscaler Client Connector (SDC) ▾

Labs 11 and 12 use the Solutions Demo Center (SDC) environment, which contains pre-populated data for incidents and reports. Before accessing the SDC Experience Center, you re-enroll the Corp: Client PC 1 using the provided IT department credentials.

  1. Log out of Zscaler Client Connector:
    • On Corp: Client PC 1, open Zscaler Client Connector.
    • Select Log Out.
  2. Log in using the IT department credentials:
    • Use the Solutions Demo Center (SDC) credentials provided for the IT_department.
    • Enter the IT_department_username.
    • Select Login.
    • At the IdP authentication prompt:
      • Enter the IT_department_username (if not prefilled), then select Next.
      • Enter the Session_Password, then select Sign in.

    Verify: After a few minutes, confirm that Internet Security Service Status shows ON and Network Type shows Off-Trusted Network.

    Zscaler Client Connector may minimise to the taskbar and take a few minutes to initialise and finish enrolling all services.

Task 12.2 View Current DLP Incidents ▾

The Incidents page in ZWA shows all DLP violations in the connected organisation. You open a specific incident and review its metadata — the overview (policy, user, severity) and the Violation Details (the actual data that triggered the policy).

  1. Access the Experience Center:
    • Log in using your SDC credentials.
    • Go to Administration > Workflow Automation > Incidents.
  2. Review incident details:
    • From the Incidents list, open the first incident by selecting its Transaction ID.
    • Review the metadata in the Overview and Violation Details sections.
  3. Examine the trigger data:
    • Scroll to the Violation Content card.
    • Expand View Trigger Data to view the content that triggered the DLP violation.

    Verify: Confirm you can view the incident metadata and the trigger data for the selected incident.

Task 12.3 Modify Incident Metadata ▾

Updating an incident's status and priority communicates triage progress to the rest of the security team. Every change is recorded in the incident's State Changes history — providing a full audit trail.

  1. Update the incident status:
    • On the Incident Details page, open the Actions menu.
    • Set Status to Investigating.
    • In the Note field, enter Test.
  2. Update the priority:
    • Select Assign Priority.
    • Change Priority to High (if it is not already set).

    Verify: Refresh your browser. Confirm the Status and Priority fields reflect your changes, then scroll to State Changes to view the history entries.

Task 12.4 Test User Notification and Escalation Workflows ▾

Notifying the incident originator gives them the opportunity to provide a business justification before the incident is escalated or closed. Escalation routes the incident to a designated approver — a manager, security lead, or compliance officer — who can make the final call.

Step A — Notify the End User

  1. From the Incident Details page, open the Actions menu, then select Notify User.
  2. In Notify User settings:
    • Select a language.
    • In the Note to the user field, enter: Please justify uploading this document for testing purposes of HandsOnLab
    • Select Submit.

    In a live environment, the originator receives an email notification similar to the sample shown above. In this lab session the email goes to the SDC demo user.

  3. Verify the incident state:
    • On the Incident Details page, review State Changes to confirm the notification was sent.

    You may proceed to the next step if the originator does not provide a justification during your lab session.

Step B — Configure an Approver and Escalate

  1. Configure an approver for escalation:
    • Go to Administration > Identity > Workflow Automation > Approvers.
    • Select Add more (+).
    • Enter your name and email address.
    • Select Save.
  2. Escalate the incident:
    • Return to the Incidents list and open the Actions menu for the Transaction ID you previously modified.
    • Select Escalate.

    Integration with SAML or SCIM allows the portal to populate the Approver list based on your organisational directory.

    • From the Select Approver dropdown list, choose your ID, then select Save.

Step C — Approve the Escalation from Email

  1. Open your personal email account and find the DLP Incident Notification email (check your spam folder if necessary).
  2. Select the Click here link in the email.
  3. Select Approved in the Justification Type field, enter a reason, then select Submit.
  4. Return to the Incident Details page.

    Verify: Confirm the notification and escalation actions appear in the incident State Changes history.

Task 12.5 Configure Automated Workflows ▾

Workflow Templates define the automated logic for incident handling — the "if this, then that" rules that replace manual triage steps. Workflow Mappings connect a template to specific incident attributes, so the right template fires automatically when the right kind of incident occurs.

Step A — View Workflow Templates

  1. In the Experience Center, go to Administration > Workflow Automation > Workflow Templates.
  2. In the Auto Escalate row, select the View Workflow Template icon in the Action column.

    The flow diagram illustrates the template logic — for example, automatically notifying a manager when an incident is created.

Step B — Explore Workflow Mappings

  1. Go to Administration > Workflow Automation > Workflow Mapping.
  2. Expand the first mapping to view the triggering attributes and conditions.

    A workflow triggers only if an incident matches the criteria defined in the mapping.

  3. Explore the available attributes and logic constructs.

    You have read-only credentials in the SDC environment — you cannot save changes. Select Reset when you are finished exploring.

Verify: Confirm you can view the workflow template logic and the workflow mapping conditions.
