ZIA Browser Isolation

Browser Isolation renders risky web content — social networking sites, unknown categories, uncategorised URLs — inside a remote Zscaler container, so no active content ever touches the endpoint. In this lab you build an Isolation Profile, enforce it via URL Filtering, and verify that browsing social sites shows the isolation watermark.

Lab 4: ZIA Browser Isolation

LAB 4

Scenario: Your organisation wants to allow access to social networking sites (X, Instagram, Facebook, LinkedIn) but prevent data exfiltration and malware download by rendering them in a remote isolated browser. You configure a Browser Isolation Profile with watermarking, then enforce it via a URL Filtering policy rule.

Time: ~20 minutes
Service: Zscaler Internet Access (ZIA)
💭 THINK FIRST · BROWSER ISOLATION

Browser Isolation sends only rendered pixels to the endpoint — no HTML, JavaScript, or files ever reach the user's browser. Before you configure it, think about what this means for the threat model and user experience.

  • Browser Isolation is described as sending "pixels only" to the user's browser. What category of attacks does this eliminate, and what is the performance/experience trade-off?
    Q1 — MODEL ANSWER
    Browser Isolation eliminates client-side code execution attacks: drive-by downloads, malicious JavaScript, browser exploits, and phishing form submissions that harvest credentials locally. Since no active content is transmitted to the endpoint, the local browser cannot execute it. The trade-off is latency — a round trip to the remote rendering container adds perceptible delay, and interactive features (rich text editors, complex JS apps) can feel sluggish. Isolation is best applied to risky-but-necessary categories rather than all web traffic.
  • You enable "Preserve Original URL" in the Isolation Profile. What user experience problem does this solve?
    Q2 — MODEL ANSWER
    Without Preserve Original URL, the browser's address bar shows the Zscaler isolation proxy URL (e.g. https://isolate.zscaler.net/…) rather than the actual site URL. Users see an unfamiliar URL, which is confusing and triggers IT support calls. Enabling Preserve Original URL rewrites the address bar to show the actual destination URL, making the isolation experience transparent — users see x.com in the address bar even though they're viewing a remote rendering.
  • The URL Filtering rule uses Action = "Isolate" rather than "Allow" or "Block". How does ZIA handle a request that matches an Isolate rule, and what happens if Browser Isolation is not licensed?
    Q3 — MODEL ANSWER
    When ZIA matches a request to an Isolate rule, it redirects the session to the Cloud Browser Isolation service, which fetches the content server-side and streams a rendered pixel view to the user's browser. If the Browser Isolation add-on is not licensed, the Isolate action falls back to Allow — the site is accessible, but without isolation. This is a deliberate "fail-open" behaviour to avoid productivity disruption when a licence lapses; admins should monitor this and not rely on Isolate as a hard block.
Task 4.1 Create a Browser Isolation Profile ▾
  1. In the Experience Center, navigate to Policies > Internet Access > Browser Isolation > Isolation Profiles.
  2. Select + Add Isolation Profile.
  3. Configure the profile with the following settings:
    • Name: isolate_profile-social_networking-internet
    • Status: Enabled
    • Preserve Original URL: Enabled
    • Watermark: Enabled
    • Watermark Content: Date & Timestamp
  4. Leave all other settings at their defaults and select Save.

Note: The watermark overlays a semi-transparent timestamp on every rendered page, making screenshots or photos of the screen attributable to a specific session. This deters exfiltration via screen capture.

Task 4.2 Create a URL Filtering Rule to Enforce Isolation ▾
  1. Navigate to Policies > Internet Access > URL & Cloud App Control > URL Filtering.
  2. Select + Add Rule.
  3. Configure the rule:
    • Rule Name: isolate-outbound-socialsite
    • Rule Order: 1 (drag to top)
    • Status: Enabled
    • URL Categories: Social Networking
    • Action: Isolate
    • Isolation Profile: isolate_profile-social_networking-internet
  4. Select Save and Activate.

Note: Rule Order 1 ensures this rule is evaluated before any existing Allow rules for social sites. If an Allow rule had a lower order number, it would match first and the Isolate rule would never fire.

Task 4.3 Test Browser Isolation on Social Networking Sites ▾
  1. On the Linux Client VM, open Chrome (ensure ZCC is running and connected).
  2. Navigate to https://x.com (formerly Twitter).
  3. Confirm that:
    • The page loads (not blocked).
    • A watermark with date and timestamp is overlaid on the page.
    • The address bar shows x.com (Preserve Original URL is working).
  4. Repeat the test for:
    • https://www.instagram.com
    • https://www.facebook.com
  5. Optionally test https://www.linkedin.com — if a SaaS App Control policy allows LinkedIn without isolation, it should open normally (no watermark). If it is also categorised as Social Networking, it will be isolated.
  6. In the Experience Center, check Analytics > Insights to verify Isolation sessions appear in the activity log.

Verify: The watermark and isolation chrome (Zscaler border or toolbar) confirm that content is being rendered in the remote container. Right-clicking the page should show limited or no standard context menu options — another sign of isolation.

Task 4.4 Exempt LinkedIn via SaaS App Control Policy ▾

The URL Filtering "Isolate" rule applies to the entire Social Networking category — this includes LinkedIn. In this task you create a SaaS App Control policy that specifically allows LinkedIn with no isolation, overriding the URL Filtering rule for that single app.

  1. Navigate to Policies > Access Control > Internet & SaaS > SaaS Application Control > Policies.
  2. Select the Add drop-down and choose Social Networking.
  3. Configure the Cloud App Control rule:
    • Rule Order: 1
    • Rule Name: allow-outbound-linkedin-any
    • Cloud Application: LinkedIn
  4. Under Action, set:
    • Viewing: Allow
    • Description: Not isolating the LinkedIn Social Networking site
  5. Select Save.
  6. Select Activate to push the policy change.
  7. Verify the behaviour:
    • Open linkedin.com — it should load normally with no isolation watermark (direct access).
    • Open x.com and instagram.com — both should still open in isolation with the watermark visible.

Why this works: SaaS App Control policies are evaluated before the URL Filtering "Isolate" rule. When ZIA matches linkedin.com to the allow-outbound-linkedin-any Cloud App Control rule (Action: Allow), it does not proceed to the Isolate URL Filtering rule for that specific app. Other Social Networking sites that lack a specific Cloud App Control rule fall through to the Isolate rule.

MENTAL NOTE
  • Pixels Only: Browser Isolation renders content server-side and streams pixels to the endpoint — no JavaScript, HTML, or files touch the local browser.
  • Two-Part Config: The Isolation Profile defines behaviour (watermark, clipboard, URL controls); the URL Filtering rule routes traffic to that profile using Action = Isolate.
  • Fail-Open: Without an Isolation licence, the Isolate action falls back to Allow — not a hard block.
  • Rule Order: Always place Isolate rules above existing Allow rules so they match first.
