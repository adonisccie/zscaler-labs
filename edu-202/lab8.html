ZDX Cloud App Monitoring

Remote users are reporting choppy audio during Teams calls. Before you can fix it, you need data. ZDX Cloud App Monitoring lets you deploy synthetic Web and Cloud Path probes that continuously measure availability and network performance from every user's endpoint to a cloud application. In this lab you build a monitoring collection for a custom mail server application using both probe types.

Lab 8: ZDX Cloud App Monitoring

LAB 8

Scenario: You are a Network Operations Center (NOC) Engineer and remote users are complaining about choppy audio during Microsoft Teams calls. You decide to configure Zscaler Digital Experience (ZDX) monitoring for a custom cloud application to gather synthetic performance data. You will create an End User Collection, add a custom Mail Server application, then configure both a Web probe (HTTP GET to measure availability and page load time) and a Cloud Path probe (ICMP to trace network hops and measure latency) running every 5 minutes from all user endpoints.

Time: ~20 minutes
Service: Zscaler Digital Experience (ZDX)
Note: Allow 15–30 min after setup for metrics to appear
💭 THINK FIRST · ZDX MONITORING

ZDX uses synthetic probes running on enrolled endpoints to continuously measure application and network performance. Before configuring them, think about what each probe type measures and when you'd use one vs. the other.

  • A Web probe performs a real HTTP GET to the target URL, measuring DNS resolution time, TCP connection time, TLS handshake time, and page load time. A Cloud Path probe sends ICMP packets hop-by-hop to the target. What is the key operational difference — and why do you configure both rather than just one?
    Q1 — MODEL ANSWER
    The Web probe answers "Is the application responding and how fast?" — it measures end-to-end availability and application-layer performance. The Cloud Path probe answers "Where in the network is the delay?" — it traces the route from client to server and measures per-hop latency and packet loss. Together they enable rapid triage: if the Web probe shows degraded page fetch time, the Cloud Path probe tells you whether the bottleneck is at the client LAN, the ISP, the Zscaler cloud edge, or the application server itself. Without Cloud Path you would only know the application is slow; with it you can point to a specific segment in the path.
  • ZDX probes run from the user's enrolled ZCC endpoint, not from a data centre probe server. Why does this matter for diagnosing remote user experience problems?
    Q2 — MODEL ANSWER
    Traditional network monitoring probes measure performance from a fixed data centre vantage point — they cannot capture Wi-Fi signal quality, home ISP congestion, or local device resource constraints at the remote user's desk. ZDX probes running on the ZCC endpoint capture the experience from the exact path the user's traffic takes: local Wi-Fi → home router → ISP → ZIA cloud edge → application. This means ZDX can surface a problem that is specific to one user's ISP, a particular Wi-Fi access point, or a device running out of CPU — issues that a centralised probe would never see because it bypasses that segment entirely.
  • You set the End User Collection Probing Criteria to "All" (all user groups, users, locations, devices). You set the Exclusion Criteria to "None". The PDF notes that if probing criteria is defined at the collection level, the probe-level criteria is ignored. What does this mean operationally for managing a large deployment?
    Q3 — MODEL ANSWER
    The collection-level Probing Criteria acts as an umbrella scope — if you define it there, all probes within that collection inherit it and cannot be overridden per-probe. This simplifies management in large deployments: define the user scope once at the collection level rather than configuring it identically for every probe. However, it removes per-probe flexibility. The best practice is to use collection-level criteria for broad scope ("monitor all users") and only configure probe-level criteria when you need a specific probe to cover a narrower segment of users than the collection default — for example, a probe that should only run for users in APAC.
Task 8.1 Create End User Collection ▾

An End User Collection groups applications and their probes under a shared scope of users and exclusions. You will create a collection called "Custom Application" that covers all users with no exclusions.

  1. Sign in to the Experience Center at console.zscaler.com.
  2. Navigate to Configuration > Digital Experience Monitoring > Configuration > Probes > End User.
  3. Select Add to create a new End User Collection.
  4. In the Add End User Collection drawer, under End User Collection Settings, enter:
    • Name: Custom Application
    • Status: Enable
  5. Under Exclusion Criteria, leave all fields set to None:
    • User Groups: None
    • Users: None
    • Location Group: None
    • Zscaler Locations: None
    • Departments: None
    • Devices: None
  6. Under Probing Criteria, set all fields to All:
    • User Groups: All User Group
    • Users: All Users
    • Location Group: All Location Groups
    • Zscaler Locations: All Zscaler Locations
    • Departments: All Departments
    • Devices: All Devices
  7. Select Add.
  8. Confirm the End User Collection named Custom Application appears in the list.

Note: If probing criteria is defined at the collection level (as it is here), that scope takes precedence — any probing criteria configured at the individual probe level will be ignored. Configure the scope at whichever level makes sense for your deployment.

Task 8.2 Create Custom Application ▾

Within the End User Collection, you add applications to monitor. You will create a custom "Mail Server" application — initially disabled, so no probes run until you configure them and then enable it.

  1. Navigate to the Custom Application End User Collection you created in Task 8.1.
  2. Select Add Application.
  3. In the Add Application drawer, enter:
    • Name: Mail Server
    • Status: Disabled (leave disabled for now — you will enable it after adding probes)
    • Application Type: Custom
  4. Select Add.
  5. Confirm the Mail Server application appears under the Custom Application collection in the left-hand tree.

Why create it disabled? The application starts Disabled so that ZDX doesn't begin probing before the probes are configured. You will enable it in Task 8.3 after the Web probe is created — at that point, ZDX will automatically activate the Web probe and begin collecting metrics.

Task 8.3 Create Custom Probes ▾

You will configure two probes for the Mail Server application: a Web probe (HTTP GET to measure availability and load time) and a Cloud Path probe (ICMP to trace the network path and measure hop-by-hop latency). Then you will enable the application so probing begins.

Step 1 — Add Web Probe

  1. Under the Mail Server application, select Add Probe.
  2. In Add Probe in Custom Application, select End User Web, then select Next.
  3. Under Configure Web Probe, enter the following values, then select Next:
    • General:
      • Name: Mail Server_Web
      • Application: Mail Server
      • Run Frequency (minutes): 5
    • Probing Criteria:
      • Users: All
      • User Group: All
      • Locations: All
      • Location Group: All
      • Departments: All
      • Devices: All
    • Exclusion Criteria:
      • Users: None
      • User Group: None
      • Locations: None
      • Location Group: None
      • Departments: None
      • Devices: None
    • Additional Parameters:
      • Destination URL: https://mail.google.com
      • Number of Attempts: 2
      • Timeout (seconds): 60
      • Follow Redirect: Enable
      • Maximum redirect: 5
  4. Review the configuration and select Submit to create the Web probe.
  5. Confirm the Mail Server_Web probe appears under Mail Server (Status: Disabled, Protocol: GET, every 5 minutes).

Step 2 — Enable the Mail Server Application

  1. In the Mail Server application panel, click the three-dot menu (⋮) and select Edit.
  2. In the Edit Application drawer, set Status to Enabled.
  3. Select Update to save.
  4. Confirm Mail Server now shows Enabled and the Mail Server_Web probe status changes to Enabled automatically.

Step 3 — Add Cloud Path Probe

  1. Under the Mail Server application, select Add Probe again.
  2. In Add Probe in Custom Application, select End User Cloud Path, then select Next.
  3. Under Configure Cloud Probe, enter the following values, then select Next:
    • General:
      • Name: Mail Server_Cloud
      • Status: Enabled
      • Application: Mail Server
      • Probe Type: Cloud Path
      • Follow Web Probe: Mail Server_Web
      • Run Frequency (minutes): 5
    • Probing Criteria:
      • Users: All
      • User Group: All
      • Locations: All
      • Location Group: All
      • Departments: All
      • Devices: All
    • Exclusion Criteria: all None
    • Cloud Path Probe Configuration:
      • Protocol: ICMP
      • Packet Count: 5
      • Interval (ms): 1000
      • Timeout (ms): 1000
      • Cloud Path Host: mail.google.com
  4. Review the probe configuration and select Submit to create the Cloud Path probe.
  5. Confirm that both probes now appear under Mail Server:
    • Mail Server_Web — Enabled, GET, every 5 minutes
    • Mail Server_Cloud — Enabled, ICMP, every 5 minutes

Important: After enabling the custom cloud application, ZDX enables the Web probe automatically. Depending on your ZDX plan, it can take 15–30 minutes for metrics to appear for a newly configured application. Continue to the next lab, and then return later to verify the metrics in ZDX Analytics.

MENTAL NOTE
  • Endpoint-Based Probes: ZDX synthetic probes run on enrolled ZCC endpoints — not central servers — to capture the real end-user network path.
  • Two Probe Types: Web probe measures application-layer availability and load time; Cloud Path probe traces the network route hop-by-hop using ICMP.
  • Diagnosis: Together they let you distinguish application-level slowness from a network-layer bottleneck and pinpoint exactly where degradation occurs.
