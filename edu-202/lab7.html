LAB 7 · ZPA

ZPA Zero Trust Browser

Zero Trust Browser (ZTB) delivers browser isolation inside ZPA — clientless users get a rendered-only view of private applications through a secure cloud browser. No files are transferred to the endpoint: copy, paste, upload, and download are all controlled by the isolation profile. In this lab you create a render-only profile and isolate the DVWA app for clientless users.

Lab 7: ZPA Zero Trust Browser

LAB 7

Scenario: Your organisation allows contractors to access the internal DVWA application through a browser without installing the Zscaler Client Connector. You need to ensure that these clientless users get a render-only view — they can see and interact with the app, but cannot download files, copy sensitive data, or upload malicious content to the endpoint. You will configure ZPA Browser Isolation with a render-only profile and validate the controls on the endpoint.

Time: ~20 minutes
Service: Zscaler Private Access (ZPA) — Browser Isolation
Prerequisite: DVWA App Segment created in Lab 6
💭 THINK FIRST · ZERO TRUST BROWSER

ZPA Browser Isolation renders the application in a cloud browser — only pixels travel to the endpoint. Before configuring it, think about the security model it enforces and how isolation policies differ from access policies.

  • In a render-only session, the application UI is streamed as pixels to the user's browser. No files, DOM content, or clipboard data leave the cloud browser. Why is this a stronger control than blocking specific file-download MIME types at a proxy?
    Q1 — MODEL ANSWER
    MIME-type blocking is a list-based control — it can be bypassed by renaming file extensions, using base64 inline encoding, or exploiting unknown MIME types. Render-only isolation is a structural control: there is no file to block because the rendering engine never sends the application's raw data to the endpoint at all. Only the visual output (pixel stream) leaves the cloud browser. Attackers cannot bypass a control that operates at the architecture level — there is no path for data exfiltration regardless of file type or encoding.
  • The Isolation Profile controls copy/paste, upload, download, and printing. The Isolation Policy controls which sessions get isolated. What happens if a user matches the Isolation Policy criteria but you set Rule Action = "Bypass Isolation" instead of "Allow Isolation"?
    Q2 — MODEL ANSWER
    "Bypass Isolation" explicitly removes the isolation requirement for sessions that match that rule's criteria — the user connects to the app directly via Browser Access without a cloud browser intermediary. This is useful for exceptions: for example, if most contractors are isolated but a specific privileged user group needs full copy/paste (e.g. to paste credentials into a legacy app), you can add a Bypass rule with higher priority that matches their user group and bypasses isolation. The Isolation Policy is evaluated top-down, so rule order matters — a Bypass rule above a general Allow Isolation rule will be matched first.
  • You set Watermark = Disabled in the Isolation Experience. In what operational scenario would you enable the watermark, and what does it protect against?
    Q3 — MODEL ANSWER
    Watermarking embeds the logged-in user's email or ID as a visible overlay on the rendered screen. Even though users cannot download files or copy data from the cloud browser, they could photograph the screen with a phone or camera. The watermark deters screen-capture exfiltration because any leaked screenshot can be traced back to the user whose session was compromised or who leaked the image. This is valuable for applications that display highly sensitive data (PII, financial records, customer data) where even a pixel-for-pixel screenshot represents a data breach risk.
Task 7.1 Create Isolation Profile ▾

An Isolation Profile defines the security controls applied during a browser isolation session — what users can and cannot do in the cloud browser. You will create a render-only profile that disables data transfer but allows viewing office files.

  1. In the Experience Center, navigate to Policies > Access Control > Clientless > Browser Isolation > Profiles.
  2. Select Add to create a new profile.
  3. Under General, enter:
    • Name: render_only_profile
    • Turbo Mode: Enabled
  4. Under Company Settings, leave all values at their defaults.
  5. Under Security, configure the controls:
    • Copy: Disabled
    • Paste: Disabled
    • Upload: Disabled
    • Download: Disabled
    • Print: Disabled
    • View Office Files in Isolation: Enabled (leave this one on)
  6. Under Regions, select Select All to allow isolation from any Zscaler data centre region.
  7. Under Isolation Experience, leave defaults:
    • Browser Experience: Native browser experience
    • Watermark: Disabled
  8. Select Save.

Task 7.2 Create Isolation Policy ▾

An Isolation Policy determines which sessions are sent to the cloud browser. You will create a policy that routes Web Browser clients accessing the DVWA app segment through the render-only isolation profile you created in Task 7.1.

  1. Navigate to Policies > Access Control > Clientless > Browser Isolation > Isolation Policy.
  2. Select Add to create a new policy rule.
  3. Configure the rule:
    • Policy name: isolate_policy_render_only
    • Rule Action: Allow Isolation
    • Isolation Profile: render_only_profile (the profile created in Task 7.1)
  4. Under Criteria, configure:
    • Client Types: Web Browser
    • Applications: Select the DVWA app segment
  5. Select Save.
  6. Verify the policy rule appears in the Isolation Policy list with Status = Enabled.

Note: The Isolation Policy evaluates against Client Types. Selecting "Web Browser" ensures the policy applies to clientless users accessing DVWA through Browser Access. Users with ZCC installed would match a different client type and could be exempted if needed.

Task 7.3 Test Render-Only Isolation ▾

With the isolation profile and policy in place, you will simulate a clientless user accessing DVWA through Browser Access and verify that the render-only controls are enforced.

Step 1 — Disconnect ZCC

  1. On Corp: Client PC 1, right-click the Zscaler Client Connector icon in the system tray and select Sign Out (or Disconnect).
  2. This simulates a clientless user — no ZCC tunnel is active, so the user must access private apps through Browser Access.

Step 2 — Access DVWA via Browser Access URL

  1. Open a browser and navigate to the DVWA Browser Access URL. You can find this in the App Segment details under the Access Method > Browser Access section — it will be in the format https://dvwa-<studentid>.labtenant.com.clientless.two.zscalerportal.net.
  2. You will be redirected to the SAML IdP for authentication. Sign in with your student admin credentials.
  3. After authentication, the DVWA application loads in the cloud browser isolation session. Confirm you can see the DVWA login page rendered in the browser with a Zscaler banner or render-only indicator.

Step 3 — Validate Render-Only Controls

  1. Log in to DVWA with credentials admin / password.
  2. Try to copy text from the DVWA page — the clipboard operation should be blocked.
  3. Try to right-click and access browser context menus — these may be restricted or limited.
  4. Navigate to DVWA Security > File Upload. Attempt to upload a file. The page should show "Protected Storage is empty" — the cloud browser has no access to the local filesystem, so no files can be selected for upload.
  5. Try to print or use Ctrl+P — print functionality should be disabled.
  6. Confirm the Zscaler isolation banner is visible on the page, indicating the session is running through browser isolation.

Expected result: Users can VIEW and interact with DVWA content through the cloud browser, but cannot download files, upload files from the endpoint, copy application data to the local clipboard, or print. The isolation profile's render-only controls are enforced at the cloud browser layer.

MENTAL NOTE
  • Pixels Only: ZPA Browser Isolation separates the rendering engine from the endpoint — only pixels reach the user's browser, enforcing data-loss controls without a client agent.
  • Profile vs Policy: The Isolation Profile defines what is restricted; the Isolation Policy defines which sessions are isolated.
  • Strictest Posture: Render-only mode (all transfer controls disabled) — users can view and interact but cannot exfiltrate data in any direction.
