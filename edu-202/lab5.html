LAB 5 · ZPA

ZPA Privileged Remote Access

Privileged Remote Access (PRA) gives contractors and third-party vendors a browser-based RDP or SSH session to internal servers — no VPN, no Zscaler Client Connector, no credentials ever revealed to the user. In this lab you build the full PRA stack: App Segments, a Privileged Portal, Consoles, Credentials, and two policies, then verify live sessions in the admin portal.

Lab 5: ZPA Privileged Remote Access

LAB 5

Scenario: You are an IT engineer tasked with enabling secure, time-limited access for third-party vendor support to troubleshoot critical Microsoft Windows and Linux servers in your organisation. These servers are inside the corporate environment and accessible only through the Zero Trust Exchange™. The organisation requires a solution that avoids VPNs and does not require the Zscaler Client Connector for external users, ensuring access is limited strictly to authorised personnel.

Time: ~50 minutes
Service: Zscaler Private Access (ZPA) — PRA
💭 THINK FIRST · PRIVILEGED REMOTE ACCESS

PRA gives browser-based, clientless access to internal servers without a VPN or ZCC. Before you build the configuration, think through why the components are separated the way they are and what security properties each one enforces.

  • PRA avoids both VPN and the Zscaler Client Connector for external users. What is the architectural reason for this, and what residual risk does a browser-based session still carry?
    Q1 — MODEL ANSWER
    VPNs grant broad network access once a user connects — a compromised contractor account can pivot laterally across the corporate network. ZCC similarly requires software installation on an unmanaged device. PRA solves both by rendering RDP or SSH in the Zscaler cloud and streaming only pixels to the browser, so no network route is opened to the internal server from the user's device. The residual risk is the authenticated browser session itself: session hijacking (if the portal URL or cookie is stolen), insecure endpoints with screen-recording malware, or social engineering into an authorised session share. This is why PRA includes session recording and join/share controls.
  • The PRA data model separates App Segments (network reachability), Portals (HTTPS entry points), Consoles (protocol + host combinations), and Credentials (injected passwords). Why might this separation matter for day-to-day operations?
    Q2 — MODEL ANSWER
    Separation lets you reuse components independently. A single App Segment can back multiple Consoles (RDP and SSH to the same host). A Portal can expose dozens of Consoles to different user groups. Credentials can be rotated in one place without touching the policy. Consoles Policies control session behaviour (clipboard, recording) independently from which credentials are injected (Credential Policies). This decomposition also follows least-privilege: a vendor account can be granted access to specific Consoles only, not to the entire App Segment or Portal.
  • Privileged Credential Injection means the vendor never sees the administrator password. What concrete threat does this mitigate compared with simply telling a contractor the password over email?
    Q3 — MODEL ANSWER
    Sharing credentials over email creates a permanent secret that exists on the contractor's email client, phone, and possibly forwarded to colleagues. It is very hard to revoke: even if the password is changed, the contractor may have reused it or stored it in a password manager. Credential injection means: (1) the credential never leaves ZPA's vault, (2) the contractor cannot extract it from the session, (3) access is revoked immediately by disabling the Credential Policy — no password change required, (4) ZPA audit logs record exactly when and how the credential was used. This shifts the trust boundary from "trusting the vendor with the password" to "trusting ZPA's access control."
Task 5.1 Create PRA Application Segments ▾

This lab uses two application segments: one for the Windows VM (RDP) and one for the Log Streaming Connector VM (SSH). You also create a single Access Policy rule that covers both.

App Segment 1 — Windows Server RDP

  1. In the Experience Center, go to Policies > Access Control > Private Applications > App Segments.
  2. Select Add Application Segment.
  3. Configure the application segment:
    • Name: PRA_RDP1_win Svr
    • Applications: 10.0.0.1 (IP address of the Corp: Win Svr)
    • Access Type: Privileged Remote Access
    • Protocol: RDP
    • Port: 3389
    • Connection Security: Any
  4. Under Client Connector Access, configure:
    • Default Port Ranges: RDP (3389)
    • TCP Port Ranges: 3389 – 3389
    • UDP Port Ranges: 3389 – 3389
  5. Under Common Configuration, keep all values as-is. Select Next.
  6. Select the Segment Group: CorpApps. Select Next.
  7. Select the Server Group: CorpServers. Select Next.
  8. Review the settings and select Save.
  9. Create the Access Policy rule. Open the Access Policy editor (either from the Save confirmation or via Policies > Access Policy). Select Add and configure:
    • Rule Name: PRA_Allow_Access
    • Rule Action: Allow Access
    • App Connector Selection Method: All App Connector Groups
    • Under Criteria, select Applications and choose PRA_RDP1_win Svr.
  10. Select Save.

App Segment 2 — Log Streaming Connector SSH

  1. Select Add Application Segment again and configure:
    • Name: PRA_SSH2_Log_Streaming_Connector_PC
    • Applications: 10.0.0.20 (IP address of the Corp: Log Streaming Connector)
    • Access Type: Privileged Remote Access
    • Protocol: SSH
    • Port: 22
  2. Under Client Connector Access, set Default Port Ranges: SSH (22).
  3. Under Common Configuration, keep values as-is. Select Next.
  4. Select Segment Group: CorpApps. Select Next.
  5. Select Server Group: CorpServers. Select Next.
  6. Review and select Save.
  7. Edit the PRA_Allow_Access policy rule and add PRA_SSH2_Log_Streaming_Connector_PC to the Application Segments criteria. Select Save.

Note: Both app segments must appear in the PRA_Allow_Access rule criteria before PRA sessions can be established. The Access Policy controls whether ZPA will proxy the traffic at all; the PRA-specific policies (Consoles Policy, Credential Policy) control session behaviour and credential injection.

Task 5.2 Create the Privileged Portal ▾

The Privileged Portal is the HTTPS URL that users access to reach their assigned Consoles. It acts as the authenticated entry point for all PRA sessions.

  1. Navigate to Policies > Access Control > Clientless > Access Methods > Privileged Portals.
  2. Select Add.
  3. Configure the following values:
    • Name: Privileged_Portal
    • Status: Enabled
    • Certificate Type: Managed
    • Portal URL: praportal (Zscaler appends the domain suffix automatically)
    • Domain Suffix: Select any available option from the drop-down menu
    • User Portal for Portal Links: None
  4. Select Save.

Note: The full portal URL will be https://praportal-<student-id>.labtenant.com.pra.two.zscalerportal.net (or similar). This URL is where users will navigate to start PRA sessions — bookmark it for Task 5.7.

Task 5.3 Create Privileged Consoles ▾

Consoles represent the devices that users can access via the portal. Each console maps a protocol (RDP/SSH) and port to a specific App Segment.

  1. Navigate to Policies > Access Control > Clientless > Resources > Privileged Consoles.
  2. Select Add.
  3. On the Console Type tab, configure:
    • Privileged Remote Access Portal: Privileged_Portal
    • Privileged Remote Access Applications: Select all application segments created in Task 5.1 — 10.0.0.1:3389 (RDP) and 10.0.0.20:22 (SSH)
  4. Select Next.
  5. On the Consoles tab, verify the details for each application:
    • Console 10.0.0.1 — Protocol: RDP, Port: 3389, Connection Security: Any, Display on Console: Enabled
    • Console 10.0.0.20 — Protocol: SSH, Port: 22, Display on Console: Enabled
  6. Select Next.
  7. Review all details and select Save.

Note: Consoles are named after the IP address / domain by default (e.g. "10.0.0.1"). The Windows icon appears on the RDP console and a terminal icon on the SSH console, making them easy to identify on the portal.

Task 5.4 Create Privileged Credentials ▾

Privileged Credentials let third-party users or contractors sign in to a console without entering credentials directly — ZPA injects them automatically at session time.

Credential 1 — Windows Server

  1. Navigate to Policies > Access Control > Clientless > Resources > Privileged Credentials.
  2. Select Add and configure:
    • Name: PRA_windows
    • Type: Username-Password
    • User Name: administrator
    • Password: Admin-123!
  3. Select Save.

Credential 2 — Linux (Log Streaming Connector)

  1. Select Add again and configure:
    • Name: PRA_Log_Streamming_Connector
    • Type: Username-Password
    • User Name: admin
    • Password: zscaler
  2. Select Save.

Note: The password is stored encrypted in the ZPA vault and never transmitted to the user's browser. Rotating the password requires only editing the credential here — no policy changes needed.

Task 5.5 Create Privileged Credential Policies ▾

Credential Policies map stored credentials to specific consoles. Without a matching Credential Policy, ZPA will not inject credentials — the user would need to type them manually.

Credential Policy 1 — RDP (Windows)

  1. Navigate to Policies > Access Control > Clientless > Privileged Remote Access > Credentials Policy.
  2. Select Add and configure:
    • Name: PRA_Credential_policy_RDP
    • Privileged Credentials: PRA_windows
    • Under Criteria, select Privileged Consoles and choose 10.0.0.1
  3. Select Save.

Credential Policy 2 — SSH (Linux)

  1. Select Add again and configure:
    • Name: PRA_Credential_policy_SSH
    • Privileged Credentials: PRA_Log_Streamming_Connector
    • Under Criteria, select Privileged Consoles and choose 10.0.0.20
  2. Select Save.

Task 5.6 Create a Consoles Policy ▾

The Consoles Policy controls session behaviour — clipboard access, recording, session join/share — for matched consoles. These controls are independent from credential injection.

  1. Navigate to Policies > Access Control > Clientless > Privileged Remote Access > Consoles Policy.
  2. Select Add.
  3. Configure the policy:
    • Name: PRA_Console_Policy
  4. Under Console Configuration, set the following:
    • Clipboard Copy: Enable
    • Record Sessions: Enable
    • Join Sessions: Enable
    • Share Sessions: Monitor
  5. On the Criteria tab, select Add Criteria → Applications.
  6. In Application Segments, select both segments from Task 5.1: PRA_RDP1_win Svr and PRA_SSH2_Log_Streaming_Connector_PC.
  7. Select Save.

Note: Share Sessions: Monitor means an admin can view the session in real time (read-only) but cannot take control. Join Sessions: Enable with Control would allow the admin to interact with the session.

Task 5.7 Test and Verify PRA Sessions ▾
  1. Log out of Zscaler Client Connector on Corp: Client PC 1. PRA is clientless — ZCC must not be running to simulate a third-party vendor accessing from an unmanaged device.
  2. On Corp: Client PC 1, sign in to the Experience Center at console.zscaler.com.
  3. Navigate to Policies > Access Control > Clientless > Access Methods > Privileged Portals.
  4. Locate Privileged_Portal and select Launch (the external link icon next to the portal URL).
  5. If prompted, log in with your student credentials: student@<Student FQDN> and your session password.
  6. The portal opens showing My Consoles (2):
    • A Windows icon tile labelled 10.0.0.1 (RDP)
    • A terminal icon tile labelled 10.0.0.20 (SSH)
  7. Select the Windows VM tile (10.0.0.1). An RDP session opens in the browser with the Windows Server desktop visible — credentials were injected automatically.
  8. Verify the session in the admin portal:
    • Go to Logs > Insights > Private Applications > Diagnostics.
    • Navigate to Privileged Remote Access > Live Sessions.
    • Confirm the active RDP session shows: Policy=PRA_Console_Policy, Host=10.0.0.1, Console Type=RDP.
  9. Return to the portal and select the SSH tile (10.0.0.20). A terminal session opens — you should be logged in as admin on the Log Streaming Connector VM.
  10. Verify the SSH session appears in Logs > Insights > Private Applications > Diagnostics > Privileged Remote Access > Live Sessions (SSH=1).

Success indicators: Both sessions open in the browser without a ZCC connection or VPN. The Live Sessions dashboard shows active RDP and SSH sessions. The vendor never typed a password — ZPA injected administrator/Admin-123! and admin/zscaler automatically.

MENTAL NOTE
  • Five Components: PRA requires an App Segment (network reach), Portal (HTTPS entry point), Consoles (protocol + host), Credentials (injected passwords), and Policies (Credential Policy + Consoles Policy).
  • Access Policy: Must explicitly allow the PRA App Segment — it is not granted automatically by the Portal configuration.
  • Browser-Based Sessions: No ZCC or VPN required — all session activity is visible under Logs › Insights › PRA › Live Sessions.
